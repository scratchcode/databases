Implementing the client

The gist

In this section we'll write the last component, the client! At the end we should have a complete system which can be ran from the command line and use it to store data.

You can see the finished code at this tag first pass complete. The code for the client is about ~20 lines and I've opted for not having tests for it for now. Testing user-interfaces tends to be more difficult than testing other pieces of code and, for now, our client is simple-enough that we can get away with not having them. The code should take only a few minutes to read and understand. If you feel like you understand the code sufficiently, move on to the next section.

Detailed explanation

There is only one commit for the client, b844e31. The way our client operates is by requiring that the user specify the name of the database in the command line and then it drops the user into a read-eval-print loop to perform database operations.

Since our database works by using two files, <db name>.keys and <db name>.values, those files will be created in whatever directory the user runs our database in (if they aren't already there). This highlights a problem with our design: it would be better to let the user specify a path to the files to use so they can be located anywhere. We would have to change a few lines of code to accommodate that, but then the user would have to specify two paths in the command line. Or maybe we could interpret the database name as a directory and put both files there. That would require that we write some more code.

The more elegant solution would be to have our database use only one file. Then the user would have to specify the path to that file, and our client could also have the convention that if only a name is given, the file is assumed to be (or be created) in the directory that the client runs. This would require that we change quite a bit of our code, though! And that's what we'll do, among a few other things, in upcoming sections.
